# PSルールとPSシミュレーター仕様

この文書は、PSルールとPSシミュレーター上の用語、評価単位、実装上の判断基準を人間向けにまとめる。

データ構造、JSONフィールド、BattleLog payload の詳細は [PS 7デッキ制シミュレーター データモデル](ps_simulator_data_model.md) を参照する。この文書はデータモデルそのものではなく、データモデルをどういうルール前提で読むべきかを説明する。

## 重要

PSシミュレーターにおいて、7デッキ全体から自由に1デッキを投げる場面は存在しない。

実戦の投げ選択は、Battle 1-3 では固定された3/2/2構成の担当枠から、Battle 4/5 では使用済みデッキを除いた残り4デッキ/3デッキから、各Battleごとの合法候補集合の中で行う。

したがって、提出案の実戦評価単位は「7デッキ集合」ではなく、「7デッキ + 3/2/2分割 + Battleごとの投げ候補」である。

この前提を外すと、固定済み7デッキ vs 固定済み7デッキの全体集合から自由に1デッキを推薦するような誤実装につながる。#112 はその混線を起こしやすいIssueであり、今後はこの文書を先に参照する。

## 1. このドキュメントの目的

この文書の目的は、PSルールとPSシミュレーター仕様を一箇所で確認できるようにすること。

特に、次の誤解を防ぐ。

- 7デッキ全体から自由に1デッキを投げると誤解する
- 7デッキ集合単体を実戦評価単位だと誤解する
- 3/2/2分割の影響を無視する
- Battleごとの合法候補集合を無視する
- `matchup_matrix` の勝率方向を誤る
- provisional deck を正式deckと誤認する
- #113 の提出案CPUが、7デッキ集合だけを評価してしまう

`docs/ps_simulator_data_model.md` は、正規化済みJSON、ID、Submission、BattleLog、Matchupなどのデータモデルを定義する文書である。この文書はその上位にあるルール・用語・判断単位の説明として扱う。

## 2. PSルール概要

PSシミュレーターのMVPは、PSの7デッキ制を前提にしている。

基本の流れは次の通り。

1. 自分側と相手側がそれぞれ7デッキを提出する。7クラスから1デッキずつで構成される。同クラスから2デッキ以上を提出することはできない。
2. 7デッキを A/B/C 担当に 3/2/2 で割り当てる。
3. Battle 1-3 は、A/B/C の担当枠ごとの候補集合から1デッキを選ぶ。
4. Battle 4-5 は、それまでに使用していないデッキだけを合法候補にして、1デッキを選ぶ。
5. 選ばれた自分側デッキと相手側デッキの勝率は `matchup_matrix` から参照する。
6. 一度使用したデッキは、勝敗に関係なく後続Battleで使えない。

現行MVPは、7デッキ提出案作成、3/2/2割当、Battle進行、使用済み制約、相性表読み込み、BattleLog、共有サマリー、provisional deck、Battle別投げ評価を扱う。

一方で、提出案そのものの自動生成、複雑な読み合い、Nash均衡、mixed strategy、正式deck/player/statusのGoogle Sheets連携は別Issueの範囲である。

## 3. 7デッキ制と提出

自分側と相手側は、それぞれ7デッキを提出する。

7クラスから1デッキずつで構成される。同クラスから2デッキ以上を提出することはできない。

7デッキは単なる自由集合ではない。提出された7デッキは A/B/C に 3/2/2 で分割されて初めて、実戦上の意味を持つ。

たとえば、同じ7デッキ集合であっても、どの3デッキをA担当へ置くか、どの2デッキをB/C担当へ置くかによって、各Battleの合法候補集合が変わる。そのため、提出案の評価では「7デッキが何か」だけでなく、「3/2/2にどう割り当てられているか」を見る必要がある。

7デッキ全体から自由に1デッキを投げる場面は存在しない。投げ評価やCPU推薦を作る場合も、7デッキ全体を候補にしてはいけない。

## 4. 3/2/2構成

提出案は、次の3担当に分かれる。

| 担当 | デッキ数 | Battle 1-3 の候補 |
| --- | ---: | --- |
| A担当 | 3 | Battle 1 / R1 |
| B担当 | 2 | Battle 2 / R2 |
| C担当 | 2 | Battle 3 / R3 |

自分側にも相手側にも、それぞれ独立した3/2/2構成がある。

3/2/2分割は提出案評価に大きく影響する。強い7デッキ集合を作っても、3/2/2分割の結果として特定Battleの候補集合が弱くなれば、実戦上の提出案としては悪くなる可能性がある。

そのため、#113 の提出案レコメンドでは「7デッキ集合」だけを評価してはいけない。評価単位は「7デッキ + 3/2/2分割 + Battleごとの投げ候補」である。

## 5. Battle / Round / R1-R5 の定義

この文書では、Battle 1-5 と R1-R5 を同じ進行単位として扱う。コードやBattleLogでは `roundNumber` として表現されることがある。

| 表記 | 候補生成 |
| --- | --- |
| Battle 1 / R1 | A担当に割り当てられた3デッキ |
| Battle 2 / R2 | B担当に割り当てられた2デッキ |
| Battle 3 / R3 | C担当に割り当てられた2デッキ |
| Battle 4 / R4 | R1-R3で使用されなかった4デッキ |
| Battle 5 / R5 | R1-R4で使用されなかった3デッキ |

R1-R3 は、固定された3/2/2担当枠から候補が決まる。R4/R5 は、担当枠ではなく使用済みデッキ制約によって候補が決まる。

候補生成のデータモデル上の詳細は [ラウンド候補生成と使用済みデッキ制約](ps_simulator_data_model.md#ラウンド候補生成と使用済みデッキ制約) を参照する。

## 6. Battleごとの投げ選択

投げ選択は、Battleごとの合法候補集合から行う。

具体的には次の通り。

- Battle 1 では、A担当3デッキの中から1デッキを選ぶ。
- Battle 2 では、B担当2デッキの中から1デッキを選ぶ。
- Battle 3 では、C担当2デッキの中から1デッキを選ぶ。
- Battle 4 では、R1-R3で使用されなかった4デッキの中から1デッキを選ぶ。
- Battle 5 では、R1-R4で使用されなかった3デッキの中から1デッキを選ぶ。

#114 / PR #117 のBattle別投げ判定は、このルールに沿っている。現行実装では `candidateDeckIdsForRound()` が各Battleの候補集合を作り、`buildBattleThrowAdvice()` がその候補集合に対して投げ評価を出す。

重要なのは、投げ評価が「現在Battleの合法候補集合」に対する評価であること。7デッキ全体からの投げ推薦ではない。

## 7. 使用済みデッキ制約

一度使用したデッキは、勝敗に関係なく後続Battleで使えない。

使用済みデッキ制約は、特にBattle 4/5の候補集合に影響する。

- Battle 4 では、R1-R3で使わなかった4デッキが候補になる。
- Battle 5 では、R1-R4で使わなかった3デッキが候補になる。

自分側の使用済みデッキと相手側の使用済みデッキは独立に管理する。自分側で使ったデッキは自分側の後続候補から除外されるが、相手側の候補生成には影響しない。

BattleLog では、各Roundの候補、選出、使用済み、残りデッキをsnapshotとして残す。後から提出案やdeck表示名が変わっても、当時のBattleを再現しやすくするためである。

## 8. `matchup_matrix` の勝率方向

`matchup_matrix` は「行デッキを使ったとき、列デッキに勝つ確率」として扱う。

たとえば、行が `deckA`、列が `deckB`、値が `0.55` の場合、意味は「`deckA` を使ったとき、`deckB` に勝つ確率が55%」である。

内部では勝率を `0.0` から `1.0` の数値として扱う。UIでは `55%` のように表示する。実装やドキュメントでは、この2つを混同しない。

自分側視点では、自分側の投げ候補を行デッキ、相手側候補を列デッキとして読む。

相手側視点で評価する場合は、視点を変える必要がある。自分側勝率 `selfWinRate` が分かっていて、同じ対面を相手側勝率として表示するなら、内部値では `1 - selfWinRate`、UI表示では `100 - selfWinRate%` を使う。

ただし、これは「同じ対面の勝率を相手側視点に直して表示する」ための変換である。未定義の逆方向 matchup を勝手に `1 - winRate` で補完する、という意味ではない。現行MVPでは、逆向き定義からの自動補完は行わない。

勝率方向を誤ると、相手CPUや相手有力候補が、相手に有利な候補ではなくこちらに有利な候補を推薦してしまう。

欠損 matchup は、現行MVPでは `0.5` fallback と warning で扱う。Battleは止めず、UI上では「データなし」件数として示す。

## 9. 自分側 / 相手側、チーム、選手

シミュレーターでは、提出案を自分側 `self` と相手側 `opponent` に分けて扱う。

自分側と相手側は、同じデータ形の `Submission` を持つ。各Submissionには A/B/C の担当、担当選手、割当デッキが含まれる。

チーム選択や選手選択は、提出案を作るためのUI上の補助である。player profile は表示品質や選手情報の見やすさを上げるための情報であり、PSルールそのものを変えるものではない。

#99 以降では、正式な deck / player / player_deck_status をGoogle Sheetsから読み込む予定がある。その場合でも、提出案評価の単位は変わらない。正式データ連携後も、7デッキを3/2/2に分割し、Battleごとの合法候補集合で投げを評価する。

## 10. `deckCandidates` / `provisionalDecks` / `unresolvedDecks`

正式deckとは、データセットの `decks` に登録され、安定した `deckId` を持つデッキである。

一方で、`matchup_matrix` の行・列には、正式deck定義にまだ存在しないデッキ名が出ることがある。#105 / PR #110 では、このような未定義デッキを session-local な仮デッキ候補として扱えるようにした。

関連する用語は次の通り。

| 用語 | 意味 |
| --- | --- |
| `deckCandidates` | APIが返す提出候補デッキ。現行では仮デッキ候補を含むことがある。 |
| `provisionalDecks` | matchup_matrix 由来で作られた仮デッキ候補。正式deck登録ではない。 |
| `unresolvedDecks` | 正式deckとして解決できなかったデッキ候補。現行APIでは仮デッキ候補と近い意味で返る。 |

provisional deck は正式登録ではない。`data/decks.csv` や正式deck定義へ自動追記しない。Google Sheetsへ書き戻すものでもない。

UI上では仮デッキとして区別する。BattleLogや共有サマリーのsnapshotでも、仮デッキであることを保持する。これにより、正式登録されていないデッキを含む対戦でも、画面やログが欠落扱いで壊れにくくなる。

将来 #99 で正式deck連携が進んでも、provisional deck は「未正式化の一時候補」として区別する。

## 11. BattleLog / snapshot

BattleLog は、Battleの進行をあとから読めるようにするための記録である。

BattleLog が記録する主な内容は次の通り。

- R1-R5の候補デッキ
- 自分側・相手側の選出デッキ
- 勝率と勝率由来
- 勝敗とスコア推移
- 使用済みデッキ
- 残りデッキ
- submission snapshot
- deck snapshot
- team / player 情報

submission snapshot は、Battle開始時点の提出案を保存する。deck snapshot は、当時のデッキ表示名、クラス、provisional flag などを保存する。

#107 / PR #111 の共有サマリーは、BattleLog の submission snapshot から7デッキ概観を表示する。これは提出案の当時状態を表示するためであり、後から正式deck名が変わっても、共有サマリー上の表示が再現しやすくなる。

BattleLog のフィールド定義は [BattleLog](ps_simulator_data_model.md#battlelog) を参照する。

## 12. 投げ評価指標

#114 / PR #117 のBattle別投げ判定では、現在Battleの合法候補集合に対して、相性表ベースの統計を出す。

主な指標は次の通り。

| 指標 | 意味 |
| --- | --- |
| 平均勝率 | 候補デッキが相手側候補集合に対して持つ平均勝率。 |
| 最低勝率 | 最も悪い対面の勝率。安定性を見る。 |
| 最高勝率 | 最も良い対面の勝率。攻撃的な上振れを見る。 |
| 5分 | 50%の対面数。 |
| 微有利 | 51%-59%の対面数。 |
| 有利 | 60%以上の対面数。 |
| 微不利 | 41%-49%の対面数。 |
| 不利 | 40%以下の対面数。 |
| データなし matchup 数 | 欠損 matchup を `0.5` fallback で扱った件数。 |

表示上の分類は次のように使う。

- 総合おすすめ: 現行実装では平均勝率を主軸にした候補。
- 安定投げ: 最低勝率が高い候補。
- 攻撃的投げ: 有利対面数や最高勝率が高い候補。
- 不利候補: 40%以下の不利対面を含む候補。
- 相手有力候補: 相手側視点で見た有力候補。

これらはすべて、Battleごとの合法候補集合に対する評価である。7デッキ全体からの投げ推薦ではない。

相手有力候補や相手自動選択は、練習・参考表示のための単純な評価である。相手の心理、癖、多段読み、mixed strategy、Nash均衡を表すものではない。

2026-06-02時点のPSシミュレーターMVPでは、主に次を扱う。

- シミュレーター用データモデル

## 14. 将来CPUとの関係

#113 は、候補デッキ群から7デッキ提出案を推薦する将来機能である。

ただし、#113 が評価すべきものは「7デッキ集合」だけではない。実戦評価単位は「7デッキ + 3/2/2分割 + Battleごとの投げ候補」である。

#113 では、候補デッキ群から7デッキを選ぶだけでなく、その7デッキをどう3/2/2に分けるか、各Battleの合法候補集合がどうなるか、使用済み制約でR4/R5に何が残るかを評価する必要がある。

#113 の先行MVPとして、相手側想定提出案CPUを検討してもよい。ただし、その場合も相手の7デッキ集合だけを出すのではなく、相手側の3/2/2分割まで含める必要がある。

#114 / PR #117 の `buildMatchupStatsForCandidate()` と `buildBattleThrowAdvice()` は、Battle単位の評価部品として後続Issueで再利用できる。

#115 は、#114 の単純なBattle別評価を超えて、minimax、mixed strategy、Nash近似、相手傾向などを検討する後続Issueである。#115 でも、評価対象はBattleごとの合法候補集合であり、7デッキ全体から自由に1デッキを投げる形式にはしない。

#99 は正式 deck / player / player_deck_status を読み込むための連携Issueである。正式データが入っても、PSルールの評価単位は変わらない。

#112 は単独Issueとして進めず、必要な考え方は #113 に吸収する。7デッキ全体から1デッキを投げるような機能は作らない。
