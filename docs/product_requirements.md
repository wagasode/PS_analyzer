# PS_analyzer プロダクト要件

## 位置づけ

PS_analyzer は、チーム内限定で利用する配信、デッキ、戦績の分析ダッシュボードです。

まずは GitHub Actions、静的ダッシュボード、Cloudflare Worker を組み合わせた現在の仕組みを整理し、短期的には限定公開化と戦績管理の基盤作りを優先します。長期的には、チーム内の分析入口として、収集、編集、確認、振り返りを一体化します。

この文書は、PS_analyzer の機能要件、非機能要件、セキュリティ要件を継続的に整理する仕様書として扱います。

#15 や #46〜#53 は現在の重点テーマとして反映していますが、それらが完了してもこの文書の役割は終わりません。新しい機能領域、運用上の制約、セキュリティ方針、MVP と将来拡張の境界が変わるたびに、実装 issue や設計 issue と合わせて更新します。

## 配信データ収集

### 目的

選手ごとの配信アーカイブを収集し、活動量、媒体、配信時間、Shadowverse 関連配信の有無を確認できるようにします。

### 要件

- YouTube と Twitch のアーカイブ情報を収集する
- 選手マスターに登録されたチャンネルを基準に収集する
- 同時配信を重複集計しない
- 配信が0件の選手も一覧に残す
- 収集できなかったチャンネルと理由を確認できる
- GitHub Actions の summary とダッシュボードで最新状態を確認できる

### 現在の位置づけ

既存の配信データ収集とレポート生成は MVP の中核です。今後は、限定公開化後も同じ確認導線を維持し、収集失敗やデータ不足が見落とされない状態を重視します。

## デッキ情報管理

### 目的

配信アーカイブと使用デッキを紐づけ、デッキ別の利用状況や確認すべき未入力データを把握できるようにします。

### 要件

- デッキ定義を安定したキーで管理する
- 配信アーカイブとデッキを紐づける
- クラス、アーキタイプ、デッキURL、補足メモを扱う
- ダッシュボード上でデッキリンクを編集できる
- 保存はブラウザのローカル操作ではなく、認証されたサーバー側経路で行う
- 保存後に再生成されたダッシュボードで結果を確認できる

### 現在の位置づけ

Save API によるデッキ情報保存は、既存の主要機能です。今後は Cloudflare Access 移行と合わせて、保存経路の保護、運用手順、失敗時の確認方法を整備します。

## 戦績管理

### 目的

試合結果、対戦相手、使用デッキ、勝敗、イベント単位の文脈を整理し、チーム内の振り返りと対策に使える状態にします。

### 要件

- 試合単位の戦績を管理する
- 自分側と相手側のデッキ、クラス、アーキタイプを扱う
- 勝敗、試合日、イベント、メモを記録する
- 期間別、選手別、デッキ別、相手別に確認できる
- 配信アーカイブやデッキ情報と将来的に紐づけられる
- 入力と閲覧はチーム内限定にする

### 現在の位置づけ

戦績管理ページは今後の重点領域です。まずは限定公開前提の設計を行い、その後に MVP として閲覧、入力、集計の最小単位を決めます。関連する設計 issue として #52 があります。

## チーム内限定公開

### 目的

配信、デッキ、戦績の情報をチーム関係者だけが閲覧できるようにし、公開URLを知っているだけでアクセスできる状態を解消します。

### 要件

- ダッシュボード閲覧を認証済みユーザーに限定する
- GitHub Pages 直URLに依存しない公開基盤へ移行する
- 既存の公開データを棚卸しする
- 直URL公開を停止、または無害化する
- チーム内で共有するURLと運用手順を明確にする

### 関連 issue

- #15: PS_analyzer全体をCloudflare Access配下へ移行する
- #46: 公開データ棚卸しと分類を行う
- #47: GitHub Pages直URLを停止・無害化する
- #48: PS_analyzerリポジトリをprivate化する影響を確認する
- #49: Cloudflare Pages + Access配信基盤を作成する
- #50: dashboard publish flowをCloudflare Pagesへ移行する

## Cloudflare Access移行

### 目的

閲覧と保存の両方を、チーム内限定の認証レイヤーで保護します。

### 要件

- Cloudflare Pages と Cloudflare Access を使った限定公開基盤を作る
- ダッシュボード公開フローを GitHub Pages から Cloudflare Pages へ移行する
- Save API Worker を Access 前提で保護する
- Access 移行後の運用手順をドキュメント化する
- 移行中も preview 確認と PR レビューの導線を維持する

### 関連 issue

- #49: Cloudflare Pages + Access配信基盤を作成する
- #50: dashboard publish flowをCloudflare Pagesへ移行する
- #51: Save API WorkerをAccess前提で保護する
- #53: Cloudflare Access移行後の運用ドキュメントを整備する

## 非機能要件

- チームメンバーが迷わず確認できること
- 収集、生成、公開の状態が追跡できること
- GitHub Actions の失敗原因を確認しやすいこと
- 手動確認が必要な箇所をダッシュボードや summary で見つけられること
- 変更は issue 単位で小さく進められること
- データ生成物と手入力データの責務が分かれていること
- MVP 段階では Python 標準ライブラリ中心の簡素な構成を維持すること

## セキュリティ要件

- チーム外部からの閲覧を制限する
- ブラウザに GitHub token などの書き込み権限を持つ secret を置かない
- Save API は repository、branch、origin、payload schema を検証する
- CORS allowlist を認証の代替として扱わない
- Cloudflare Access などの認証レイヤーを公開面の前段に置く
- repository の private 化や Pages 停止の影響を事前に確認する
- 公開済みデータの棚卸しと無害化を移行計画に含める

## MVPと将来拡張

### MVP

- 配信アーカイブの収集
- 選手別、チーム別の集計
- 配信タイムラインの確認
- デッキ定義と配信アーカイブの紐づけ
- ダッシュボード上でのデッキ編集
- Save API 経由の repository data 更新
- チーム内限定公開への移行計画と基盤作成
- 戦績管理ページの設計

### 将来拡張

- 戦績入力、編集、集計ページ
- 配信、デッキ、戦績を横断した分析ビュー
- 大会、期間、対戦相手単位の比較
- 未入力、要確認、確認済みのワークフロー管理
- Access 移行後の利用者別運用ルール
- データ品質チェックとアラート
- 長期的な履歴分析とシーズン比較
